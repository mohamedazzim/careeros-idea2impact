# Architecture Decision Record: ADR-002 — Why Qdrant for Vector Embeddings Store

## Status
Approved

## Context
RAG operations in CareerOS AI require high-throughput similarities comparisons, low latency queries, dense payload filters, and dynamic namespace partitioning to support thousands of active candidates' custom resume packages without cross-tenant memory leakage.

## Decision
Qdrant was chosen as the primary semantic vector intelligence database for the following reasons:

1. **Sub-millisecond Search Latency**: Written in Rust, Qdrant utilizes SIMD hardware acceleration and highly optimized HNSW (Hierarchical Navigable Small World) graphs to compute p50 similarity searches under 5 milliseconds.
2. **Robust Payload Filtering Co-processing**: Allows SREs to apply single-stage metadata filters (e.g. `userId == ... AND status == 'active'`) DURING the vector similarities comparison, preventing recall decay caused by post-filtering.
3. **In-Memory and On-Disk Auto-Scaling**: Efficiently handles memory constraints by offloading immutable dense vectors to Mmap (memory-mapped files) on disk, keeping only the search graphs in RAM.
4. **Dynamic Snapshot backups**: Synchronous logically compliant snapshot creation enables zero-downtime hot-backups meeting our structural RPO targets.

## Consequences
- **Positive**: Ultra-fast horizontal scaling, rigorous multi-tenant data segregation, and high-performance Rust execution limits.
- **Negative**: Demands structured RAM allocations during heavy indexing events and custom vector configuration mappings on setup.
