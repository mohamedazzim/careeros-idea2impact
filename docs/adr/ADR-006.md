# Architecture Decision Record: ADR-006 — Why Redis Memory Layer for Enterprise Tokens Caching

## Status
Approved

## Context
High concurrent traffic from candidates causes repeated user session checks, identical vector match queries, and rate-limiting lookups, threatening to slow down the database and saturate key-value stores.

## Decision
An in-memory Redis replication layer is approved to protect critical backend services:

1. **Sub-millisecond Key-Value Store Actions**: Runs directly on memory, processing over 100,000 read operations per second under 1 millisecond.
2. **Native Key Expiration (TTL)**: Automatically expires cached sessions and temporary validation tokens, preventing data bloat.
3. **Redis-backed Token Buckets for Rate Limiting**: Employs efficient atomic transaction keys (`INCR` and `EXPIRE`) to enforce strict API access parameters.
4. **Temporary Session Sharing Cache**: Decouples API services by holding transient user states, making our instances fully stateless and autoscalable.

## Consequences
- **Positive**: Substantially reduced PostgreSQL indexing loads, instant rate limiting, and under-1ms session validations.
- **Negative**: Increases infrastructure complexity and requires dual database write syncs to prevent caching drift.
