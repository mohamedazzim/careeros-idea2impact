# Architecture Decision Record: ADR-007 — Why Celery for Distributed Async Workers Queue

## Status
Approved

## Context
AI-powered document generation, multi-stage graph traversals, and PDF scanning are slow, resource-heavy operations. Running them directly inside Express/FastAPI HTTP threads would block the event loop, causing connection timeouts and server crashes.

## Decision
Celery is approved as our asynchronous task queue and distributed worker manager:

1. **Clean Event-Driven Decoupling**: Completely frees HTTP request loops by immediately registering long-running tasks and routing execution payloads to separate worker nodes.
2. **Broad Broker compatibility (Redis/RabbitMQ)**: Integrates seamlessly with our Redis cache layer to manage high-throughput task queues.
3. **Resilient Retry & Error Containment**: Provides native task-level retry guidelines, timeout policies, and error handling to ensure failing runs don't affect other operations.
4. **Dynamic Scaling Workers**: Allows developers to scale compute nodes horizontally based on real-time task queue volumes.

## Consequences
- **Positive**: Reliable, fast server responses, clear progress tracking, and resilient background task execution.
- **Negative**: Adds deployment complexity, requiring separate monitoring daemons (such as Flower) and queue container setups.
